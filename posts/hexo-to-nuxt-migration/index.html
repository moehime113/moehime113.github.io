从 Hexo 到 Nuxt：博客迁移全记录

为什么要离开 Hexo？

Hexo + Butterfly 这套组合颜值没得说，Butterfly 的设计审美我很喜欢。但随着博客越折腾越复杂，一些痛点逐渐暴露出来。

PJAX 与 PWA 的相爱相杀

Butterfly 内置了 PJAX 局部刷新和 PWA 离线缓存。单独看每个都是好东西，合在一起就变成了玄学——写完文章 push 上去，PWA 缓存不更新，读者看到的永远是旧版。无痕模式打开是新的，正常模式死活不刷新。清理 Service Worker、硬刷新、改缓存策略，折腾一圈发现根本原因很简单：生成的静态页面本质上是黑盒，你很难精确控制缓存行为。

构建速度与生态

文章数量上来之后，hexo g 的时间肉眼可见地变长。Hexo 的机制是全量生成，每次都要把所有页面重新渲染一遍。插件生态虽然丰富，但质量参差不齐，主题和插件之间的耦合经常让人头疼。

为什么选 Nuxt？

看过 VitePress、Astro、Next.js，最终选了 Nuxt 4 SSG。核心考量：

HexoNuxt 4
构建机制CLI 全量生成静态文件Nitro 引擎预渲染，增量构建
模板/组件EJS/Pug，无组件化Vue 3 组件，自动导入
内容管理front-matter + markdown@nuxt/content v3，类型安全查询
交互内联 script，靠 CDN完整的 Vue 响应式生态
路由配置文件 _config.yml文件系统路由，动态参数
暗色模式主题配置 + 手动脚本localStorage + CSS 变量，SSR 安全

最大的吸引力：内容格式不变

@nuxt/content v3 直接读 markdown，frontmatter 字段几乎不用改。这意味着迁移成本大头不在内容，只在工程搭建。所有文章原封不动搬过来就能用。

Vue 生态的开发体验

组件化意味着每个 UI 单元是独立的、可复用的、可测试的。useAsyncData + queryCollection 这套数据获取模式比 Hexo 的 _config + front-matter + theme 散落配置清晰太多。写交互不需要 hack，不需要往 HTML 里塞 <script> 标签。

迁移过程

内容层：零改动搬过来

所有 markdown 文件从 Hexo 的 source/_posts/ 复制到 Nuxt 的 content/posts/。frontmatter 稍微做了统一：

Hexo:        title, date, tags, categories, cover
Nuxt:        title, date, description, categories, cover

多了个 description 字段用于 SEO 和列表摘要，去掉了 tags（暂时没用）。内容正文一个字没改。

路由：从 permalink 到文件系统

Hexo 的 URL 靠 _config.yml 里的 permalink 模式定义。Nuxt 直接用文件系统路由：

Hexo:  /posts/hexo-setup/
Nuxt:  /posts/hexo-to-nuxt-migration/

区别在于 Nuxt 路由由 pages/ 目录结构决定，posts/[slug].vue 处理所有文章。配置了 trailingSlash: true 保证 URL 一致性。404 用 [...slug].vue 兜底。

设计系统：从 Stylus 变量到 CSS 自定义属性

Hexo + Butterfly 用 Stylus 预处理器定义主题变量，迁移之后直接用 CSS 自定义属性，不需要重新编译样式，主题切换是即时的。

部署：保持一致

CI 沿用 GitHub Actions，部署流程和原来基本一样。

架构对比

Hexo 的架构

Hexo CLI
  ├── _config.yml        # 全局配置
  ├── themes/butterfly/  # 主题（模板 + 样式 + 脚本）
  │   ├── _config.yml   # 主题配置
  │   ├── layout/       # EJS 模板
  │   └── source/       # 静态资源
  ├── source/_posts/     # 文章 markdown
  └── public/            # 生成输出

Hexo 是生成器模式：CLI 命令触发，读取配置 + 文章 + 主题，全量渲染成 HTML。主题和内容是分离的，但数据流是隐式的——配置散落在 _config.yml、_config.butterfly.yml、frontmatter 三个地方，模板里用不同的变量名访问。

Nuxt 的架构

Nuxt App
  ├── pages/             # 文件系统路由 → SSG 预渲染
  ├── components/        # Vue 组件（自动导入）
  ├── content/           # Markdown 内容（@nuxt/content）
  │   ├── posts/        # 博客文章集合
  │   └── pages/        # 独立页面集合
  ├── composables/       # 可组合逻辑（useTheme, useSite）
  ├── assets/css/        # 设计系统（tokens + base）
  └── Nitro 引擎
      ├── 预渲染器       # crawlLinks 自动发现页面
      └── 输出 .output/public/  # 纯静态文件

Nuxt 是框架模式：内容、组件、路由、构建都被统一管理。数据流是显式的——queryCollection 查询内容，useAsyncData 管理异步数据，useTheme() 封装主题逻辑。每一层职责清晰，不会出现 Hexo 那种"配置在 A 处，变量在 B 处，效果在 C 处"的情况。

数据流对比

环节HexoNuxt
内容获取读文件 → 模板注入变量queryCollection('posts').all()
数据管理无，模板即数据useAsyncData + 响应式
站点配置_config.yml + 主题 configuseSite() composable
主题Stylus 变量 → 编译到 CSSCSS 自定义属性 → 运行时切换
交互状态全局变量 + DOM 操作Vue 响应式 + composables

预渲染策略

Hexo 每次 hexo g 都是全量重新生成所有页面。文章多了之后，这个时间变得不可忽视。

Nitro 的 crawlLinks: true 则是从 seed routes 出发，按 <a href> 链接爬取需要预渲染的页面。只渲染能找到的、被引用的页面。但要注意：crawlLinks 不能替代显式 routes 声明。如果你的导航是客户端 JS 动态生成的，或者某些页面没有直接的 <a> 链接指向，crawlLinks 可能漏掉它们。所以在 nitro.prerender.routes 里显式声明关键路由是个好习惯：

prerender: {
  crawlLinks: true,
  routes: ['/', '/about', '/posts', '/links'],
}

迁移成本远低于预期

说实话出发前我还挺忐忑的，担心工程量会很大。结果发现 markdown 内容零改动，核心工作量只在：

  1. 搭建 Nuxt 项目骨架
  2. 设计 CSS token 系统
  3. 写几个核心组件

借助claude code加在一起两天就搞定了。框架升级的收益是长期的，成本是一次性的。

总结

迁移之前，在折腾 Hexo 的各种"魔改"时经常会想——"要是能直接用 Vue 写就好了"。Node.js 生态的 SSG 方案已经非常成熟，对于博客这种内容驱动的站点，用 Nuxt + @nuxt/content 几乎是最佳实践。

如果你的博客内容主要是 markdown，而你又有点前端基础，迁移到 Nuxt 绝对值得。 内容不用动，开发体验质变，交互能力完全释放。
